## 1. 代码总览——文件在哪里

### 1.1 Java 服务的包结构

每个 Java 服务的源码都在 `services/{服务名}/src/main/java/com/codec/kb/{模块}/` 下。

| 服务目录 | 对应功能 | 主要包名 |
|---|---|---|
| `services/kb-common/` | 公共工具库，被其他服务引用，不独立运行 | `com.codec.kb.common` |
| `services/kb-gateway/` | 唯一对外入口：鉴权、管理后台、对话路由 | `com.codec.kb.gateway` |
| `services/kb-doc-service/` | 文档上传、解析、分块 | `com.codec.kb.doc` |
| `services/kb-index-service/` | 知识库索引、向量存储、混合检索 | `com.codec.kb.index` |
| `services/kb-ai-service/` | RAG 编排、LLM 调用、会话持久化 | `com.codec.kb.ai` |
| `services/kb-eureka-server/` | 服务注册中心（可选） | `com.codec.kb.eureka` |

### 1.2 kb-gateway 内部子包

网关是最复杂的服务，内部按功能拆成若干子包：

| 子包 | 文件举例 | 负责什么 |
|---|---|---|
| `gateway/auth/` | `AuthController`, `DbUserDetailsService`, `AppUserPrincipal`, `BootstrapAdminRunner` | 注册、登录、Spring Security 鉴权 |
| `gateway/admin/` | `AdminUsersController`, `AdminDocumentController`, `AdminProvidersController`, `AdminModelsController`, `AdminTenantController`, `AdminMessageController` | 管理员后台所有操作 |
| `gateway/kb/` | `UserKbController`, `UserKbProvisioningService` | 用户知识库管理、文档上传代理 |
| `gateway/messages/` | `MessageController` | 用户留言 |
| `gateway/models/` | `ModelsController`, `AiModelResolverService`, `ProviderEmbeddingService` | 可用模型列表、模型解析 |
| `gateway/clients/` | `InternalDocClient`, `InternalIndexClient` | 调用 doc-service 和 index-service 的 HTTP 客户端 |
| `gateway/crypto/` | `CryptoService` | AES-GCM 加密/解密 |
| `gateway/store/` | `AppUserEntity`, `TenantEntity`, `AiProviderEntity`, `AiModelEntity` 等 | JPA 实体和数据库仓库（Repository） |

> **什么是 Entity / Repository？**
> Entity 是一个和数据库表一一对应的 Java 类，比如 `AppUserEntity` 对应 `app_user` 表，每个字段对应一列。Repository 是操作这张表的工具，提供 `findById`、`save`、`delete` 等方法，不需要手写 SQL。

### 1.3 Python 服务

| 文件 | 服务 | 作用 |
|---|---|---|
| `services/kb-embedder/app/main.py` | kb-embedder (8090) | 加载 BAAI/bge-small-zh-v1.5 模型，对外提供 `POST /embed` 接口，输入文本列表，返回 512 维向量列表 |
| `services/kb-ocr-service/app/main.py` | kb-ocr-service (8070) | 调用 Tesseract，对外提供 `POST /ocr` 接口，输入 Base64 图片，返回识别文本 |

## 2. 系统启动与初始化

**涉及文件**：`gateway/auth/BootstrapAdminRunner.java`

系统第一次启动时，需要有一个管理员账号和一个默认租户，否则谁也登录不进去。`BootstrapAdminRunner` 就是负责这件事的——它实现了 Spring 的 `ApplicationRunner` 接口，意味着应用每次启动后会自动运行一次。

**启动时做了什么：**

1. **确保默认租户存在**（`ensureDefaultTenant`）
   - 检查 ID 为 `00000000-0000-0000-0000-000000000001` 的租户是否存在
   - 不存在就创建一个，名称是"默认组织"，邀请码是"DEFAULT"

2. **确保管理员账号存在**（`ensureAdminUser`）
   - 读取环境变量 `KB_BOOTSTRAP_ADMIN_USER` 和 `KB_BOOTSTRAP_ADMIN_PASSWORD`
   - 如果没有配置且允许不安全默认值（`KB_ALLOW_INSECURE_DEFAULTS=true`），就用 `admin` / `admin123`
   - 如果管理员已存在但状态不是 ACTIVE，自动修复为 ACTIVE
   - 如果不存在就创建，角色设为 ADMIN，状态设为 ACTIVE，关联默认租户

## 3. 用户注册与审批

**涉及文件**：
- `gateway/auth/AuthController.java` — 处理注册请求
- `gateway/store/AppUserEntity.java` — 用户数据表结构
- `gateway/store/TenantRepository.java` — 查询租户（用于邀请码验证）
- `gateway/admin/AdminUsersController.java` — 管理员审批、禁用、恢复操作
- `gateway/kb/UserKbProvisioningService.java` — 审批通过后分配知识库

### 3.1 注册流程

请求：`POST /api/auth/register`

```
前端发送请求体：
{
  "username": "alice",
  "password": "user123",
  "displayName": "Alice",
  "inviteCode": "ACME2024"   ← 可选
}
```

`AuthController.register()` 做了以下事情：

1. **参数校验**：用户名格式（字母数字 3-32 位）、密码长度（6-72 位）、头像格式（仅限 PNG/JPG/WebP/GIF 的 Data URL）
2. **重名检查**：查询数据库，若用户名已存在返回 409 错误
3. **处理邀请码**：若填写了 `inviteCode`，查找对应租户并获取其 ID；若找不到或租户已禁用，归入默认租户
4. **创建用户**：密码经 BCrypt 加密后保存，状态设为 `PENDING`（待审批）
5. **返回**：`{ "ok": true, "status": "PENDING" }`

### 3.2 管理员审批

请求：`POST /api/admin/users/{id}/approve`

`AdminUsersController.approve()` 做了以下事情：

1. 查询用户是否存在（不存在返回 404）
2. 若用户已是 ACTIVE，直接返回成功（幂等）
3. 将用户状态改为 ACTIVE，保存到数据库
4. 调用 `UserKbProvisioningService.ensureDefaultKb(userId)` 为用户分配默认知识库（确保登录后能看到至少一个知识库）

### 3.3 管理员禁用 / 恢复

新增请求：

- `POST /api/admin/users/{id}/disable`
- `POST /api/admin/users/{id}/restore`

规则：

1. 只有超管可以禁用 / 恢复
2. 禁用只允许 `ACTIVE -> DISABLED`
3. 恢复只允许 `DISABLED -> ACTIVE`
4. 禁用不会删除用户数据，也不会清除其租户归属、知识库绑定或租户管理员标记
5. 管理员账号和当前登录管理员本人都不能被禁用

登录影响：

- `AuthController.login()` 会区分两类 `403`
- 若账号是 `DISABLED`，返回“您的账号已被禁用，请联系管理员”
- 若账号仍是 `PENDING` / `REJECTED`，保留“账号未启用或待管理员审批”

## 4. 登录与鉴权

**涉及文件**：
- `gateway/auth/AuthController.java` — 登录接口
- `gateway/auth/DbUserDetailsService.java` — Spring Security 的用户加载器
- `gateway/auth/AppUserPrincipal.java` — 登录用户的身份对象
- `gateway/SecurityConfig.java` — 鉴权规则配置

### 4.1 Spring Security 是怎么工作的

每一个进来的 HTTP 请求，Spring Security 都会拦截，判断这个请求是否有权限访问目标接口。判断依据是请求头里的凭据：

- **Basic Auth**：请求头 `Authorization: Basic BASE64(用户名:密码)`，Spring Security 自动解码并调用 `DbUserDetailsService.loadUserByUsername()` 查库验证
- **Session Cookie**：先调用 `POST /api/auth/login` 建立 Session，后续请求自动带上浏览器的 Cookie，Spring Security 从 Session 中读取用户信息

### 4.2 DbUserDetailsService 做了什么

`loadUserByUsername(username)` 被 Spring Security 在需要验证身份时自动调用：
1. 用用户名（大小写不敏感）查询数据库 `app_user` 表
2. 找不到就抛出 `UsernameNotFoundException`，Spring Security 返回 401
3. 找到了就返回一个 `AppUserPrincipal` 对象

### 4.3 AppUserPrincipal 是什么

这是登录用户在内存中的身份对象，Spring Security 会把它存起来，之后代码里随时可以通过 `AuthUtil.requirePrincipal()` 取到。

它包含：用户 ID、用户名、密码哈希、显示名、角色（USER/TENANT_ADMIN/ADMIN）、账号状态、租户 ID、是否是租户管理员。

还有一个重要方法 `getAuthorities()`，它根据角色返回权限列表，例如 ADMIN 用户对应 `ROLE_ADMIN` 权限，而 `SecurityConfig` 里的规则 `.hasRole("ADMIN")` 就是靠这个判断的。

### 4.4 鉴权规则在哪里配置

`SecurityConfig.java` 里的 `filterChain` 方法定义了所有路径的访问规则：

| 路径模式 | 访问要求 |
|---|---|
| `/api/health`、`/api/health/live`、`/api/auth/**` | 所有人可访问（匿名） |
| `/api/admin/**` | 必须有 `ROLE_ADMIN` |
| `/api/tenant/**` | 必须有 `ROLE_ADMIN` 或 `ROLE_TENANT_ADMIN` |
| `/api/messages/**` | 任意已登录用户 |
| `/api/**` 其他 | 任意已登录用户 |

## 5. 文档上传

文档上传有三种模式：单文件、批量、断点续传。三种模式都经过网关转发给 doc-service 处理。

### 5.1 文件路由路径

```
前端 → kb-gateway (UserKbController / AdminDocumentController)
           ↓ 转发（携带 X-Internal-Token）
       kb-doc-service (DocIngestController / ResumableUploadController)
           ↓ 解析分块后调用
       kb-index-service (IndexController)
```

### 5.2 单文件上传

**网关侧**：`gateway/kb/UserKbController.java`

- 接口：`POST /api/kbs/{kbId}/documents/upload`
- 权限检查：调用 `requireOwnedKb(uid, kbId)` 确认该知识库属于当前用户
- 通过后将请求转发给 doc-service

**doc-service 侧**：`doc/DocIngestController.java`

`uploadToKb()` 方法执行完整的解析入库流程（详见第 6 节）。

**关键参数约束**：
- 文件名最长 120 字符
- 标签最多 20 个，每个标签最长 40 字符
- 支持的分隔符：逗号、中文逗号、分号、空格

文件落盘路径：`{KB_DOC_DATA_DIR}/uploads/{kbId}/{docId}/{filename}`

### 5.3 批量上传

**接口**：`POST /api/kbs/{kbId}/documents/batch`（用户）或 `POST /api/admin/kbs/{kbId}/documents/batch`（管理员）

doc-service 的 `batchUpload()` 方法遍历每个文件，逐一调用与单文件相同的解析入库逻辑。每个文件独立返回状态：

```json
{
  "results": [
    { "filename": "a.txt", "docId": "doc_xxx", "status": "ok" },
    { "filename": "b.pdf", "status": "error", "error": "parse failed" }
  ]
}
```

**ZIP 导入**：`importZip()` 解压 ZIP 文件，过滤掉 `__MACOSX` 目录和以 `.` 开头的隐藏文件，对其余文件逐个走单文件解析流程。

### 5.4 断点续传

**涉及文件**：
- `doc/ResumableUploadController.java` — HTTP 接口层
- `doc/ResumableUploadStore.java` — 文件状态和数据管理

断点续传分三步：

**第一步：初始化**（`POST /api/kbs/{kbId}/uploads`）

客户端告诉服务器文件名、总大小、Content-Type，服务器返回 `uploadId` 和分块大小（固定 4MB）。

`ResumableUploadStore.initUpload()` 做的事：
- 在 `{KB_DOC_DATA_DIR}/resumable/{kbId}/{uploadId}/` 创建目录
- 创建 `meta.json`（记录文件名、总大小、已接收字节数、已覆盖的字节范围）
- 预分配 `data.bin`（大小等于总文件大小的空文件）

**第二步：上传分片**（`PATCH /api/kbs/{kbId}/uploads/{uploadId}`）

客户端携带请求头 `Content-Range: bytes 0-4194303/10485760` 和二进制数据。

`ResumableUploadStore.applyChunk()` 做的事：
1. 解析 Content-Range，获取起始和结束位置
2. 检测重叠：如果这段字节已经上传过，直接跳过（支持客户端重试）
3. 用 `RandomAccessFile` 在 data.bin 的指定位置写入数据
4. 更新 `meta.json` 记录已覆盖的字节范围（支持乱序上传）
5. 当所有字节都覆盖完毕，标记状态为 `COMPLETE`，自动触发解析入库

**第三步：查询进度**（`GET /api/kbs/{kbId}/uploads/{uploadId}`）

返回 `{ "received": 已接收字节数, "total": 总字节数, "status": "UPLOADING/COMPLETE/INGEST_ERROR" }`

## 6. 文档解析与分块

**涉及文件**：
- `doc/ParserRegistry.java` — 解析器工厂，按文件类型选择解析器
- `doc/parsers/` 目录下各解析器 — 具体的文本提取逻辑
- `doc/Chunker.java` — 将长文本切成若干小块
- `doc/OcrClient.java` — 调用 OCR 服务处理图片

### 6.1 解析器是怎么选的

`ParserRegistry.parseOrNull(file, contentType)` 按以下顺序依次判断哪个解析器支持当前文件，第一个支持的就用它：

| 解析器 | 支持格式 | 底层库 |
|---|---|---|
| `TextParser` | `.txt` `.log` `.csv` | 直接读文本 |
| `MarkdownParser` | `.md` `.markdown` | 直接读文本 |
| `HtmlParser` | `.html` `.htm` | 提取纯文本 |
| `DocxParser` | `.docx` | Apache POI |
| `PdfParser` | `.pdf` | Apache PDFBox |
| `ExcelParser` | `.xls` `.xlsx` | Apache POI |
| `ImageOcrParser` | `.png` `.jpg` 等图片 | 调用 kb-ocr-service |

如果所有解析器都不支持，返回 `null`，该文件被跳过（不报错，但也不入库）。

### 6.2 OcrClient 怎么调用 OCR 服务

`OcrClient.ocr(bytes, filename, contentType)` 把图片的二进制数据 Base64 编码后，发送给 kb-ocr-service：

```
POST {KB_OCR_URL}/ocr
{
  "filename": "photo.png",
  "contentType": "image/png",
  "dataBase64": "iVBORw0KGgo..."
}
```

返回：`{ "text": "识别出的文字内容" }`

如果没有配置 `KB_OCR_URL`，`OcrClient.enabled()` 返回 false，图片文件被跳过。

### 6.3 分块算法

`Chunker.chunk(text)` 将一段长文本按固定窗口切成若干块：

**参数**：`maxChars = 900`，`overlapChars = 120`（这两个值在 DocIngestController 中硬编码）

**算法步骤**：
1. 正规化空白字符（去掉多余的空行和空格）
2. 按段落分割
3. 对每个段落：
   - 如果 ≤ 900 字符，直接作为一个块
   - 如果 > 900 字符，用滑动窗口切割：每次取 900 字符，下一个窗口从第 `900 - 120 = 780` 字符处开始，与上一个块有 120 字符的重叠
4. 合并过小的块：相邻的小于 120 字符的块会被合并成一个

**为什么要有重叠？** 避免一个句子刚好被切在两个块的边界，导致检索时两个块都匹配不完整。

## 7. 向量嵌入

向量嵌入是把一段文字转换成一组数字（512 个浮点数组成的数组），使得语义相近的文字在数学上也接近。这是语义搜索的基础。

**涉及文件**：
- `index/EmbeddingFacade.java` — 根据知识库配置选择嵌入模式
- `index/LocalEmbeddingBackend.java` — 本地模式策略接口
- `index/DjlLocalEmbeddingBackend.java` — 本地模式默认实现：JVM 内 DJL + ONNX Runtime
- `index/HttpLocalEmbeddingBackend.java` — 本地模式备选实现：HTTP 调用 Python kb-embedder
- `index/EmbeddingApiClient.java` — API 模式：调用外部 OpenAI 兼容接口
- `index/ManagedEmbeddingClient.java` — 托管模式：通过网关调用已配置的嵌入模型

### 7.1 三种嵌入模式

`EmbeddingFacade.embed(kbId, texts)` 根据知识库的 `embeddingMode` 字段选择对应客户端：

**模式一：`local`（本地嵌入）**

根据 `KB_EMBEDDER_MODE` 选择一个本地后端：
- `djl`（默认）：在 `kb-index-service` JVM 内通过 DJL + ONNX Runtime 直接运行模型，由 `DjlLocalEmbeddingBackend` 实现，无需外部服务。
- `http`：调用独立部署的 kb-embedder 服务：
```
POST http://localhost:8090/embed
{ "texts": ["文字1", "文字2"] }
→ { "vectors": [[0.1, 0.2, ...], [0.3, 0.4, ...]] }
```
由 `HttpLocalEmbeddingBackend` 实现。Docker Compose / K8s 部署默认使用此模式。

两种后端都使用 BAAI/bge-small-zh-v1.5（CLS 池化 + L2 归一化），输出 512 维向量，结果可互换。

**模式二：`api` + 普通模型名（直连外部 API）**

直接调用外部 OpenAI 兼容的 embedding 接口：
```
POST {embeddingBaseUrl}/embeddings
Authorization: Bearer {apiKey}
{ "model": "text-embedding-ada-002", "input": ["文字1", "文字2"] }
```
由 `EmbeddingApiClient` 实现。为了兼容用户填写的 URL 可能带或不带 `/v1`，会同时尝试 `baseUrl` 和 `baseUrl/v1` 两个地址。

**模式三：`api` + UUID（托管嵌入）**

当 `embeddingModel` 字段是一个 UUID 时，说明用户选的是后台"模型管理"里已配置的模型。此时不直接调用外部，而是让网关代劳：
```
POST http://localhost:8080/internal/embeddings
X-Internal-Token: {token}
{ "modelRef": "f8b6...", "texts": ["文字1"] }
```
由 `ManagedEmbeddingClient` 实现，网关的 `InternalEmbeddingController` 接收后，通过 `ProviderEmbeddingService` 找到该模型的真实 baseUrl/apiKey，再调用外部。

好处是：Provider 的 apiKey 只存在网关侧，index-service 完全不需要知道任何密钥。

### 7.2 向量存在哪里

知识库初始化时，向量会先以文本形式（`[0.1,0.2,...]`）落到 `idx.chunks.embedding`。随后 Flyway 会把该列升级为 pgvector 原生 `vector` 类型，最终检索始终基于原生向量列执行，不再假设固定 `512` 维。这是因为云端托管 embedding 模型可能返回 `4096` 维向量。

另外，写入阶段现在是 fail-fast：如果 embedding 请求失败，`IndexService` 会直接抛错并回滚本次入库，不再把 chunk 以空向量形式写入数据库。

## 8. 知识库混合检索

**涉及文件**：
- `index/IndexService.java` — 检索编排（调用 BM25 + 向量检索，融合结果）
- `index/HybridIndex.java` — 内存 BM25 索引的构建和计算
- `index/IndexController.java` — 暴露 `POST /internal/search` 接口

### 8.1 什么是 BM25

BM25 是一种基于词频的经典文本相似度算法，不需要向量化。它统计每个词在文档中出现的次数（TF）以及有多少文档包含这个词（DF），计算出一个相关性分数。词越罕见、在文档中出现次数越多，分数越高。

### 8.2 混合检索的完整步骤

`IndexService.search(SearchRequest)` 执行以下步骤：

**第一步：向量检索**（Dense Search）

1. 调用 `EmbeddingFacade.embed(kbId, [query])` 把用户的问题转成向量
2. 用 pgvector 的余弦相似度在数据库中找出最相近的前 `topK*2` 个块
3. 记录每个块的向量相似度分数

**第二步：BM25 检索**（Sparse Search）

1. 从内存缓存（`ConcurrentHashMap`）取出该知识库的 `HybridIndex` 对象
2. 如果缓存不存在（比如服务刚重启），从数据库加载所有分块重建 BM25 索引
3. 把用户问题分词，计算与每个块的 BM25 分数

**第三步：分数融合**

两种方法得到的分数量纲不同，先各自归一化（除以最大值变成 0-1 区间），再加权融合：

```
最终分数 = 0.65 × 向量相似度 + 0.35 × BM25分数
```

权重配置在 `application.yml` 的 `index.hybrid` 部分（`wDense=0.65`，`wBm25=0.35`）。

**第四步：排序返回**

按最终分数从高到低排列，取前 `topK` 条返回给调用方。

### 8.3 BM25 分数计算公式

`HybridIndex.scoreBm25()` 实现标准 BM25 公式：

```
对每个查询词 t：
  IDF = log((N - df_t + 0.5) / (df_t + 0.5) + 1)
  tf_part = f_t × (k1 + 1) / (f_t + k1 × (1 - b + b × 文档长度/平均长度))
  总分 += IDF × tf_part

参数：k1=1.2（词频饱和度），b=0.75（文档长度归一化）
N = 总分块数，df_t = 包含词 t 的分块数，f_t = 词 t 在本分块中的频次
```

### 8.4 BM25 索引缓存

BM25 索引完全在内存中，服务重启后缓存为空。第一次检索请求时会触发重建（从数据库加载所有分块），可能略慢。之后文档更新时缓存自动失效，下次检索再重建。

## 9. RAG 对话（非流式）

RAG 是 Retrieval-Augmented Generation（检索增强生成）的缩写：先检索知识库找到相关片段，再把这些片段连同用户问题一起送给大模型，让模型基于这些资料回答。

**完整调用链**：

```
前端 POST /api/chat
    ↓
kb-gateway: ChatController.chat()
    ↓ 转发 POST /internal/chat
kb-ai-service: AiController.chat()
    ↓ 调用
RagService.chat()
    ├─ 检索 → IndexClient → kb-index-service /internal/search
    └─ LLM 调用 → OpenAiChatClient → 外部大模型 API
```

### 9.1 ChatController（网关层）做了什么

`ChatController.chat()` 是用户请求的第一站，负责权限和配置解析：

1. **确认当前用户**：调用 `AuthUtil.requirePrincipal()` 取出登录用户的 UUID
2. **解析可用知识库**：
   - 调用 `UserKbProvisioningService.listAccessibleKbIds(userId)` 获取用户有权访问的所有知识库 ID
   - 如果请求里指定了 `kbIds`，取两者的交集；如果没指定，用全部可访问的知识库
3. **解析每个知识库的 topK**：从 `KbSettingsEntity` 读取每个知识库配置的 `documentCount`，作为该知识库的检索数量上限（默认 6，范围 1-50）
4. **解析模型**：如果请求里传了 `model`（UUID），调用 `AiModelResolverService.resolveLlm()` 查出该模型的 baseUrl/apiKey/modelId；如果传了不存在的模型，直接返回 400
5. **转发**：把处理好的参数打包成内部请求，附上 `X-Internal-Token` 和 `X-User-Id` 两个请求头，发给 ai-service

### 9.2 RagService（ai-service 层）做了什么

`RagService.chat(userId, chatRequest)` 执行主要业务逻辑：

**第 1 步：会话处理**
- 如果请求里没有 `sessionId`，或者 sessionId 对应的会话不存在，自动创建新会话
- 从数据库加载最近 `contextSize` 条历史消息（`contextSize=0` 表示不携带历史）

**第 2 步：知识库检索**
- 调用 `searchKnowledge(kbIds, message, topK, kbTopK)` 批量检索
- 检索结果按"kbId:chunkId"去重，最多保留 `MAX_CONTEXT_CHUNKS = 80` 条

**第 3 步：降级判断**
- 如果所有知识库都检索失败：返回"知识库检索服务暂时不可用"
- 如果检索成功但没有结果：返回"知识库中暂无相关信息"

**第 4 步：构建提示词**

系统提示词（给大模型的"角色设定"）：
```
你是一个企业知识助手。请严格根据提供的知识片段回答用户问题。
规则：
1) 只能使用知识片段中的信息，不得编造。
2) 如果片段不足以回答，请明确说明知识库暂无相关信息。
3) 回答中如引用片段内容，请用 [1][2] 这样的编号标注。
```

用户提示词（每次对话的问题）：
```
知识片段：
[1] {来源提示}
{片段内容（截断至 1200 字符）}

[2] ...

用户问题：{用户的问题}
请给出简洁、准确的回答，并在需要时标注引用编号。
```

**第 5 步：LLM 调用**
- 优先使用请求里传入的 `llm` 配置（baseUrl + apiKey + modelId）
- 其次使用环境变量 `SILICONFLOW_*` 配置的默认模型
- 调用 `OpenAiChatClient.chatWithMessages()` 发送请求，消息数组格式：`[系统, ...历史消息, 当前用户问题]`

**第 6 步：持久化**
- 把用户消息和模型回答都写入数据库（`ai.message` 表）
- 更新会话的 `updatedAt` 时间戳，使其在会话列表中排到最前

**第 7 步：返回**
```json
{
  "sessionId": "ses_xxx",
  "answer": "BM25 是一种...",
  "answerHash": "a0d9...",
  "citations": [
    { "chunkId": "chk_xxx", "snippet": "...", "filename": "bm25.txt", "kbId": "kb_xxx" }
  ]
}
```

### 9.3 OpenAiChatClient 怎么调用大模型

`OpenAiChatClient.chatWithMessages(baseUrl, apiKey, model, messages)` 发起一个标准的 OpenAI API 请求：

```
POST {baseUrl}/chat/completions
Authorization: Bearer {apiKey}
{
  "model": "Qwen/Qwen2-7B-Instruct",
  "messages": [
    { "role": "system", "content": "..." },
    { "role": "user", "content": "..." }
  ]
}
```

为兼容不同格式的 baseUrl，会自动尝试 `baseUrl/v1` 和 `baseUrl` 两个地址，第一个成功的返回结果。

## 10. SSE 流式对话

SSE（Server-Sent Events）是一种服务器向浏览器持续推送数据的技术。流式对话的效果是：大模型每生成一个字/词，立刻推送到前端显示，不需要等全部生成完再返回。

**完整调用链**：

```
前端 GET /api/chat/stream?message=...
    ↓
kb-gateway: ChatController.chatStream()
    ↓ 转发 POST /internal/chat/stream
kb-ai-service: AiController.chatStream()
    ↓ 调用
RagService.chatStream()
    ├─ 同步完成：知识库检索 + 提示词构建
    └─ 流式调用大模型，逐 token 写出 SSE 事件
```

### 10.1 网关层怎么转发 SSE

`ChatController.chatStream()` 解析查询参数（`message`、`kbs`、`model` 等），完成知识库和模型解析后：

1. 设置响应头：`Content-Type: text/event-stream`、`Cache-Control: no-cache`
2. 调用 ai-service 的 `POST /internal/chat/stream`
3. 将上游的 SSE 字节流**直接透传**给客户端（`transferTo()`）

### 10.2 ai-service 层怎么生成 SSE 流

`RagService.chatStream()` 前半部分与非流式完全相同（知识库检索、提示词构建），区别在 LLM 调用：

`OpenAiChatClient.streamChat()` 向大模型发送 `"stream": true` 的请求，然后逐行读取响应：

大模型每次返回类似：
```
data: {"choices":[{"delta":{"content":"BM25"}}]}
data: {"choices":[{"delta":{"content":" 是"}}]}
data: {"choices":[{"delta":{"content":"一种"}}]}
data: [DONE]
```

每读到一行，就提取 `choices[0].delta.content` 中的文字，写成 SSE 事件发送给客户端：
```
event: token
data: BM25

event: token
data:  是
```

流结束后，写出最后一个 `event: meta` 事件，包含 sessionId、answerHash 和完整的 citations 列表。

## 11. 会话管理

**涉及文件**：
- `ai/SessionService.java` — 会话和消息的数据库操作
- `ai/SessionEntity.java` — 会话表（`ai.session`）的 JPA 实体
- `ai/MessageEntity.java` — 消息表（`ai.message`）的 JPA 实体
- `gateway/SessionController.java` — 对外暴露的会话管理接口

### 11.1 会话和消息的数据结构

**会话**（`ai.session` 表）：
- `id`：会话 ID，格式 `ses_xxxxxx`（由 `IdUtil.newId("ses")` 生成）
- `userId`：会话所属用户的 UUID
- `title`：会话标题（最长 24 字符，自动截断）
- `createdAt` / `updatedAt`：每次对话后更新 `updatedAt`，用于排序

**消息**（`ai.message` 表）：
- `sessionId`：关联的会话 ID
- `sender`：`USER` 或 `ASSISTANT`
- `content`：消息内容
- `model`：生成该消息使用的模型 ID（用户消息为 null）
- `createdAt`：消息创建时间

### 11.2 用户隔离

`SessionService` 所有方法都接受 `userId` 参数。`safeTenant(userId)` 会验证 userId 是否是合法的 UUID 格式，不合法的统一映射为 `"public"`，防止越权访问其他用户的会话。

### 11.3 网关如何代理会话请求

`SessionController` 不做业务逻辑，只转发请求到 ai-service，附带 `X-User-Id` 和 `X-Internal-Token` 两个请求头。ai-service 的 `AiController` 接收后调用 `SessionService` 完成操作。

## 12. 多租户与邀请码

**涉及文件**：
- `gateway/admin/AdminTenantController.java` — 超管创建租户、管理邀请码
- `gateway/admin/AdminUsersController.java` — 授予/撤销租户管理员身份
- `gateway/auth/AuthController.java` — 注册时处理邀请码
- `gateway/store/TenantEntity.java` — 租户表

### 12.1 租户和邀请码怎么工作

**创建租户**（`AdminTenantController.create()`）：
1. 校验 `slug`（唯一短名，格式 `[a-z0-9][a-z0-9-]{0,62}`，全局唯一）
2. 生成邀请码：用 `SecureRandom` 生成 12 字节随机数，Base64 URL-safe 编码（无填充），结果类似 `aBcD_eFg-H123`
3. 如果生成的邀请码碰巧已存在，递归重新生成
4. 保存租户，邀请码随响应返回给管理员

**用户注册时使用邀请码**（`AuthController.register()`）：
- 如果填了 `inviteCode`：查询 `tenant` 表，找到匹配且 `enabled=true` 的租户，将其 `tenantId` 写入用户记录
- 如果没填或找不到：用默认租户 ID（`00000000-0000-0000-0000-000000000001`）

**轮换邀请码**（`AdminTenantController.rotateCode()`）：生成新邀请码覆盖旧的，旧码立即失效。

### 12.2 租户管理员如何工作

- **授予**（`AdminUsersController.grantTenantAdmin()`）：将用户的 `isTenantAdmin` 字段设为 `true`。用户的 `role` 字段保持 `USER` 不变，但 `AppUserPrincipal.getAuthorities()` 会额外返回 `ROLE_TENANT_ADMIN` 权限
- **撤销**（`AdminUsersController.revokeTenantAdmin()`）：将 `isTenantAdmin` 设为 `false`

这样设计是为了兼容历史数据（早期直接用 `role=TENANT_ADMIN` 记录），现在统一用 `role=USER` + `isTenantAdmin=true` 表示。

## 13. Provider 与模型管理

**涉及文件**：
- `gateway/admin/AdminProvidersController.java` — Provider CRUD、模型同步
- `gateway/admin/AdminModelsController.java` — 模型启停
- `gateway/admin/ProviderModelSyncService.java` — 从远端同步模型列表
- `gateway/models/AiModelResolverService.java` — 对话时解析模型配置
- `gateway/store/AiProviderEntity.java` — Provider 表
- `gateway/store/AiModelEntity.java` — 模型表

### 13.1 Provider 的 BaseUrl 规范化

用户填写的 baseUrl 格式可能五花八门，`AdminProvidersController` 中的 `normalizeBaseUrl()` 会自动处理：
- 去掉末尾的 `/`
- 去掉常见的路由后缀（`/chat/completions`、`/embeddings` 等）
- 如果末尾是 `/api`，补充为 `/api/v1`
- 最终结果必须以 `http://` 或 `https://` 开头

### 13.2 从 OpenRouter 同步模型

`ProviderModelSyncService.syncItems()` 处理从远端 `/models` 接口拉取的模型列表：
- 解析每个模型的 ID 和显示名称
- 查询本地数据库是否已有该模型（按 `providerId + modelId` 匹配）
- **新模型**：创建记录，`enabled = false`（默认禁用，需要管理员手动启用）
- **已有模型**：如果显示名称需要更新就更新

### 13.3 对话时如何解析模型

当用户在对话请求里传 `model: "f8b6..."` 时，`AiModelResolverService.resolveLlm()` 执行：
1. 把 `model` 值解析为 UUID（如果不是合法 UUID 返回 null）
2. 在 `ai_model` 表查找该 UUID，必须是已启用的模型
3. 查找关联的 Provider，必须是已启用的 Provider
4. 解密 Provider 的 apiKey（详见第 14 节）
5. 规范化 Provider 的 baseUrl
6. 返回 `LlmConfig(baseUrl, apiKey, modelId)`，传给 ai-service 调用大模型

还会检查该模型是否是聊天模型（通过 `ModelTagSupport.isEmbeddingModel()` 排除嵌入模型，通过 `isRerankModel()` 排除重排序模型）。

## 14. API 密钥加密

**涉及文件**：`gateway/crypto/CryptoService.java`

Provider 的 API Key 是敏感信息，不能明文存数据库。`CryptoService` 使用 AES-GCM 对称加密算法，密钥来自环境变量 `KB_CRYPTO_MASTER_KEY`。

**加密**（`encrypt(plaintext)`）：
1. 随机生成 12 字节的 IV（初始化向量），每次加密都不同，确保即使同一段文字加密两次，结果也不同
2. 用主密钥（经 SHA-256 派生为 256 位 AES 密钥）加密明文
3. 返回格式：`"v1:" + Base64(IV + 密文 + GCM认证标签)`，总长度约为原文的 1.5 倍

**解密**（`decrypt(encrypted)`）：
1. 检查是否以 `"v1:"` 开头（向后兼容：不以 `"v1:"` 开头的直接当明文返回）
2. Base64 解码，前 12 字节是 IV，剩余是密文+标签
3. 解密并验证完整性，失败返回空字符串

**注意**：如果 `KB_CRYPTO_MASTER_KEY` 没有配置，会使用硬编码默认值 `"dev-master-key-change-me"`。这在生产环境中是严重安全问题，启动时会有警告，但不会阻止启动。

## 15. 用户留言系统

**涉及文件**：
- `gateway/messages/MessageController.java` — 用户侧接口
- `gateway/admin/AdminMessageController.java` — 管理员侧接口
- `gateway/store/UserMessageEntity.java` — 留言表（`public.user_message`）

### 15.1 用户发送留言

`MessageController.sendMessage()` 处理 `POST /api/messages`：
1. 获取当前用户 ID
2. 校验：`subject` 不能为空且不超过 200 字符，`content` 不能为空
3. 创建 `UserMessageEntity`，状态设为 `OPEN`
4. 保存到数据库，返回消息 ID

### 15.2 管理员处理留言

`AdminMessageController` 提供三个操作：

**查看所有留言**（`GET /api/admin/messages`）：
- 可按 `status` 过滤（`OPEN` / `REPLIED` / `CLOSED`）
- 按创建时间倒序，显示所有用户的留言

**回复留言**（`POST /api/admin/messages/{id}/reply`）：
- 更新留言的 `reply` 字段、`repliedBy`（管理员 ID）、`repliedAt`（时间）
- 自动将状态改为 `REPLIED`

**修改状态**（`PATCH /api/admin/messages/{id}`）：
- 状态只能是 `OPEN` / `REPLIED` / `CLOSED` 之一
- 可用来把已处理的留言标记为 `CLOSED`

### 15.3 用户查看回复

`MessageController.listMyMessages()` 查询当前用户自己的所有留言，返回包含 `reply` 和 `repliedAt` 字段，用户可以在聊天页看到管理员的回复内容。
