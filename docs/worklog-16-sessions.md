# KB Assistant 项目分析与 16 次工作记录

## 说明

当前仓库只有 1 次初始化提交，无法直接还原出真实的 16 段开发历史。因此，下面的工作记录是基于现有代码结构、功能模块、数据库迁移、部署脚本和项目文档进行逆向整理后的“阶段化工作记录”。这份内容适合用于课程设计、毕设答辩、项目汇报或周报材料。

## 项目分析

### 1. 项目定位

该项目是一个面向课程设计和毕设演示的企业知识库 RAG 微服务系统，目标不是做最复杂的生产平台，而是做出“可运行、可演示、可交付”的完整方案。

### 2. 架构特征

- 后端采用 `Spring Boot 3.2 + Java 21 + Maven 多模块`。
- 前端采用 `Next.js 15 + React 19 + pnpm workspace`。
- AI 辅助能力采用 `Python FastAPI`，分别承担 embedding 与 OCR。
- 系统整体采用 `gateway + doc-service + index-service + ai-service + embedder (+ optional ocr/eureka)` 的拆分方式。
- 数据层使用一个 PostgreSQL 实例，通过 `public / idx / ai` 三个 schema 进行隔离。

### 3. 功能范围

项目已经覆盖了一套较完整的知识库问答系统能力：

- 用户注册、登录、审批、禁用恢复、个人资料维护
- 多租户、邀请码、租户管理员
- 系统知识库、全体知识库、用户私有知识库
- 文档上传、批量导入、ZIP 导入、断点续传
- 多格式文档解析、文本分块、OCR、向量化、混合检索
- 同步问答、SSE 流式问答、会话管理、引用溯源
- Provider 管理、模型同步、模型启停、密钥加密
- 管理后台、首页、聊天页三套前端
- Docker Compose 与 Kubernetes 两套交付路径

### 4. 工程化判断

这个仓库的完成度已经明显超过“单体 demo”阶段，优点是模块清楚、文档齐全、部署路径明确，适合作为展示型项目。当前也有几个明显特征：

- Git 历史较弱，仓库中只有初始化提交，过程可追溯性不足。
- 仓库中保留了运行数据、前端构建产物和 `node_modules`，更偏演示仓库而不是精简源码仓库。
- 文档较完整，说明作者更重视交付和答辩呈现效果。

## 16 次阶段性工作记录

### 第 1 次工作记录：项目调研与需求拆解

2026年3月2日~2026年3月8日(第1周)

本次主要围绕课程设计场景梳理需求，明确系统不是单纯聊天机器人，而是一个带知识库、带权限、带后台、可部署演示的企业知识库问答平台。确定了三类核心角色：普通用户、租户管理员、系统管理员。

本阶段输出了整体目标边界：系统需要支持文档入库、知识检索、问答对话、用户管理和成果展示，同时要保留较好的演示效果，便于答辩时从功能、架构和部署三个角度进行说明。

### 第 2 次工作记录：总体架构与仓库骨架搭建

2026年3月9日~2026年3月15日(第2周)

本次完成仓库总体结构设计，确定后端采用 Maven 多模块，前端采用 pnpm workspace，多语言能力通过 Python 服务补齐。目录划分为 `services`、`apps`、`packages`、`scripts`、`deploy`、`docs` 和 `data`，为后续分模块开发打下基础。

同时确定了系统采用微服务拆分方式，由 `kb-gateway` 统一对外暴露入口，`kb-doc-service` 负责文档处理，`kb-index-service` 负责索引与检索，`kb-ai-service` 负责 RAG 编排，`kb-embedder` 与 `kb-ocr-service` 作为辅助服务提供 AI 能力。

### 第 3 次工作记录：公共模块与网关基础能力建设

2026年3月16日~2026年3月22日(第3周)

本次优先建设 `kb-common` 公共模块，沉淀跨服务复用的 DTO、工具类、异常处理和内部调用鉴权过滤器，避免多个服务之间重复定义协议。通过这一层实现了服务间统一的数据契约和 `X-Internal-Token` 内部鉴权机制。

与此同时，初步搭建网关服务基础能力，包括健康检查接口、统一配置入口、静态资源承载能力以及后续对外 API 的统一接入点，明确网关作为系统唯一公网入口的职责。

### 第 4 次工作记录：用户注册登录与身份认证实现

2026年3月23日~2026年3月29日(第4周)

本次完成用户注册、登录、登出、身份查询等基础鉴权能力，实现 `POST /api/auth/register`、`POST /api/auth/login`、`GET /api/auth/me` 等核心接口。系统同时支持 `Basic Auth` 和 `Session Cookie` 两种认证方式，满足接口调试和浏览器页面两类场景。

为保证首次启动可用，新增管理员自举逻辑，在系统启动时自动创建默认租户和管理员账号，解决“系统部署后无人可登录”的问题，保证演示链路可闭环。

### 第 5 次工作记录：用户审批、账号状态治理与个人资料维护

2026年3月30日~2026年4月5日(第5周)

本次补充了完整的账号治理流程。新注册用户默认进入待审批状态，管理员审批后方可正常使用系统。与此同时，支持对账号进行禁用与恢复，保证后台具备基础的治理能力。

在用户体验层面，还增加了昵称和头像维护能力，使项目不再停留在纯接口 demo，而是具备更完整的业务系统感。该阶段也进一步细化了用户状态与角色权限边界。

### 第 6 次工作记录：知识库基础模型与默认知识库建设

2026年4月6日~2026年4月12日(第6周)

本次开始搭建知识库主线能力，在索引服务中建立知识库、文档、分块等核心数据模型，并在系统中始终保留一个默认知识库 `公司章程` 作为所有用户可见的系统级知识库，保证系统上线后即具备最小可用内容。

同时，网关侧增加用户知识库访问与绑定逻辑，为后续的私有知识库、公共知识库和租户共享知识库打下结构基础，也为权限隔离预留了接口边界。

### 第 7 次工作记录：文档上传链路打通

2026年4月13日~2026年4月19日(第7周)

本次围绕文档入库的第一步，打通了“前端/网关 -> 文档服务”的上传链路。普通用户和管理员都可以通过网关发起上传请求，再由网关统一转发到 `kb-doc-service` 处理，确保内部服务不直接暴露给外部。

在文档服务内部，建立了原始文件落盘规则，按 `kbId/docId` 管理上传文件，为后续的解析、索引和问题排查提供稳定的存储基础。

### 第 8 次工作记录：多格式解析、文本分块与索引入库

2026年4月20日~2026年4月26日(第8周)

本次重点放在文档内容处理能力上。项目完成了 `txt`、`csv`、`md`、`html`、`docx`、`pdf`、`xls/xlsx` 等多种文档格式解析，并建立统一解析注册机制，提升扩展性。

解析后的文档内容按固定窗口进行分块，采用 `900` 字符窗口和 `120` 字符重叠的策略，兼顾检索效果与上下文连续性。分块结果与文档元数据随后写入索引服务，形成可检索的数据基础。

### 第 9 次工作记录：向量化能力与混合检索实现

2026年4月27日~2026年5月3日(第9周)

本次完成 embedding 与检索能力建设。项目通过 Python `kb-embedder` 服务提供本地向量化接口，并在索引服务中封装多种 embedding 策略，兼容本地模型、外部 API 和后台托管模型三种模式。

在检索层面，系统采用 BM25 与 dense vector 的混合检索方案，默认权重为 `0.35 / 0.65`。这一设计避免纯关键词检索或纯向量检索的单一短板，更适合中文知识库问答场景。

### 第 10 次工作记录：RAG 问答主链路实现

2026年5月4日~2026年5月10日(第10周)

本次实现 AI 服务主链路，由 `kb-ai-service` 负责接收问题、调用索引服务获取候选片段、组装 prompt、请求大模型并返回答案。至此，系统从“知识库管理平台”升级为“可问答的知识库系统”。

为了增强可解释性，回答结果中同时返回引用片段信息，支持前端展示来源文档与命中内容，提升了系统在答辩和演示中的说服力。

### 第 11 次工作记录：流式对话与会话持久化

2026年5月11日~2026年5月17日(第11周)

本次进一步优化问答体验，增加 SSE 流式输出能力，使回答可以逐 token 返回，提升交互流畅度。网关负责转发流式响应，AI 服务负责调用流式大模型接口并在结束时补充元信息。

同时建立会话和消息持久化能力，将聊天记录写入 `ai` schema，支持用户继续查看历史会话、保留上下文和生成更完整的交互链路。

### 第 12 次工作记录：Provider 接入、模型同步与密钥加密

2026年5月18日~2026年5月24日(第12周)

本次实现后台的模型与 Provider 管理能力。管理员可以录入 OpenAI-compatible Provider，拉取 `/models` 接口中的模型列表，并在后台进行模型启停控制。

考虑到 API Key 属于敏感信息，系统引入密钥加密机制，采用 `KB_CRYPTO_MASTER_KEY` 对密钥进行加密存储，避免明文保存在数据库中，提升项目的安全性和工程可信度。

### 第 13 次工作记录：多租户、邀请码与租户管理员功能实现

2026年5月25日~2026年5月31日(第13周)

本次扩展出多租户能力，使项目从单租户知识库系统升级为可区分组织边界的知识库平台。管理员可以创建租户并生成邀请码，用户注册时填写邀请码后即可自动归属对应租户。

与此同时，系统加入 `TENANT_ADMIN` 角色，使部分管理动作可以下放到租户层面。知识库也进一步区分为系统知识库、全体知识库、租户共享知识库和用户私有知识库，业务模型更加完整。

### 第 14 次工作记录：用户留言系统与后台治理能力完善

2026年6月1日~2026年6月7日(第14周)

本次补充了“用户联系管理员”的闭环功能。普通用户可以提交留言，管理员可以在后台查看、回复和关闭留言，形成平台内部的反馈与处理机制。

这一阶段也同步完善了管理后台的用户管理、知识库管理、文档管理、租户管理和模型管理能力，使后台不只是展示页面，而是真正可操作的治理端。

### 第 15 次工作记录：首页、聊天页、管理后台三套前端整合

2026年6月8日~2026年6月14日(第15周)

本次完成三套前端应用建设，分别对应首页、聊天界面和管理后台。前端基于 Next.js 15 与 React 19 实现，并通过 `packages/shared` 复用公共鉴权和 CSRF 工具，降低重复开发成本。

同时，项目设计了前端构建发布链路：执行根目录构建脚本后，会将 `apps/home`、`apps/chat`、`apps/admin` 的构建产物复制到网关静态资源目录中，由网关统一对外提供页面访问。

### 第 16 次工作记录：部署交付、联调验证与文档收尾

2026年6月15日~2026年6月21日(第16周)

本次聚焦项目交付层。项目补充了 `ensure-postgres.ps1`、`setup-python.ps1`、`build.ps1` 等本地脚本，降低环境搭建门槛；同时提供 Docker Compose 和 Kubernetes 两套部署路径，满足本地演示与集群部署需求。

最后阶段完善了 `README.md`、`ARCHITECTURE.md`、`API.md`、`代码导读.md`、`DEPLOYMENT-MATRIX.md` 等文档，形成较完整的交付资料。至此，项目具备了从源码、功能、部署到答辩材料的完整闭环。

“本项目按照‘需求梳理 -> 微服务拆分 -> 鉴权与知识库基础 -> 文档入库与检索 -> RAG 对话 -> 多租户与后台治理 -> 前端整合与部署交付’的路径分阶段完成，最终形成了一套可运行、可演示、可部署的企业知识库问答系统。”
